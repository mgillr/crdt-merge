### 5 Irritants Detected (Shannon Protocol)

**IRRITANT 1: Binary Merkle for Model Parameters**

The current design imposes a binary Merkle tree on tensor data. But model parameters have natural hierarchy: model → layers → attention heads/FFN blocks → weight matrices → elements. A binary tree ignores this structure, producing deeper trees (depth ~30 for 1B params) with semantically meaningless subtree boundaries.

**Impact:** Deeper traversal = more hash comparisons. Meaningless subtree = deltas without semantic coherence.

**Fix: Model-Aware Merkle Tree** — branching factor follows model architecture. Top levels = model layers (branching: 32-128), mid levels = within-layer structure, bottom levels = parameter elements. Depth drops from ~30 to ~4-6. Deltas have semantic meaning: "layer 3, attention heads 2-4 changed" vs "leaves 47392-48901 changed".

For non-model payloads: fall back to generic binary Merkle. This is payload-adaptive, not payload-specific.

---

**IRRITANT 2: Backward Hash Compatibility**

The "default-on" design means every hash becomes H(data ‖ trust_context). In single-node mode, trust_context = {self: 1.0}. Therefore: H(data ‖ 1.0) ≠ H(data). Wire format is not backward compatible for hash comparison between E4 and pre-E4 nodes.

**Impact:** Mixed clusters can't verify Merkle roots across version boundaries.

**Fix: Compatibility Hash Mode** — E4 nodes compute BOTH hashes. When communicating with pre-E4 peers (detected via handshake), use H(data). When communicating with E4 peers, use H(data ‖ trust). Cost: ~2x hash computation for mixed-version comms. Benefit: zero-downtime migration.

---

**IRRITANT 3: Trust Inflation**

Over time, all well-behaved peers converge toward maximum trust. When everyone has max trust, trust provides zero discrimination — equivalent to having no trust system. The trust lattice becomes vacuous.

**Impact:** Long-running clusters lose Byzantine protection through trust inflation.

**Fix: Trust Homeostasis** — Inspired by biological homeostatic plasticity. Total trust budget is conserved: sum of all trust scores in any dimension = N (number of peers). When one peer's trust increases, all others decrease proportionally. High trust is RELATIVE, not absolute. Always discriminates.

Implementation: normalize trust vectors after every observation cycle. Normalization is a CRDT-compatible operation (it preserves the lattice partial order within each node's view — convergence is maintained because all nodes apply the same normalization).

---

**IRRITANT 4: Four Separate Proofs in PCO**

The current PCO bundles four independent proofs: integrity (Merkle path), causality (clock snapshot), trust attestation, minimality (delta bounds). Four separate objects = more wire overhead, more verification steps.

**Impact:** Unnecessary complexity and bandwidth.

**Fix: Aggregate PCO** — Single cryptographic signature over the composition of all four properties:

```
aggregate = H(merkle_root ‖ clock_state ‖ trust_score ‖ delta_bounds)
pco = sign(aggregate, originator_key)
```

Verification: one signature check + four property derivations from known state. Reduces PCO from 4 signatures to 1 signature + 4 hash checks. ~4x faster verification, ~75% smaller wire format.

---

The `set_default_implementation()` registry pattern hides dependencies, makes testing harder, and creates global mutable state.

**Impact:** Harder to test, harder to reason about.

### Novel Innovation: Adaptive Immune Verification (WILDCARD → SURVIVED MURDER BOARD)

**The Insight:** The biological immune system doesn't verify every cell equally. Self (trusted) cells get minimal scrutiny. Foreign (untrusted) cells trigger the full immune cascade. Only anomalous behavior from trusted cells escalates.

**Applied to E4:**

```
Level 0 (Established Peer, Trust > 0.8):
  → Verify signature only
  → Apply delta immediately
  → Merkle consistency checked lazily (next sync cycle)
  → Cost: O(1)

Level 1 (Known Peer, Trust 0.4-0.8):
  → Verify signature + Merkle root consistency
  → Apply delta
  → Full PCO verified async
  → Cost: O(log n) 

Level 2 (New/Low-Trust Peer, Trust < 0.4):
  → Full PCO verification before application
  → All four properties checked synchronously
  → Cost: O(k log n)

Level 3 (Quarantined Peer, Trust < 0.1):
  → Delta rejected without processing
  → Cost: O(1) (rejection)
```

**Why this is safe:** Even at Level 0, the signature is checked (can't forge origin). Merkle inconsistency from a rogue high-trust peer is detected at next sync (seconds to minutes). Trust drops → future deltas escalate to Level 1 or 2. The damage window is bounded by sync frequency.

**Why this is a breakthrough:** For a cluster of N established peers doing frequent syncs (the common case), verification cost drops from O(k log n) per delta to O(1) per delta. **Orders of magnitude faster for the common path.** Full verification only triggers for the uncommon path (new peers, suspicious activity).

**Murder Board Result:** A patient Sybil builds trust to Level 0 over weeks, then sends malicious deltas. Damage window: one sync cycle (~seconds). Detection: Merkle root mismatch at next sync with ANY other honest peer. Recovery: trust drops to Level 2, all subsequent deltas get full verification. **Bounded damage, automatic recovery.** Survived.

**Formal characterization:** Projection Delta is a method for computing irredundant join decompositions of distributed state using hierarchical hash-based change detection, achieving O(k log n) complexity where k is the number of changed elements and n is the total state size.

   - Depth for 1B params: ceil(log₂₅₆(1B)) = ceil(3.58) = **4 levels**

   - Threshold: configurable, default = 2σ from rolling mean

5. **Add adaptive verification depth to trust gate**
   - Four levels based on peer trust score
   - Level 0 (>0.8): signature only → O(1)

   - Level 2 (<0.4): full PCO → O(k log n)

6. **Replace service locator with constructor injection + defaults**
